# State Transition Audit

Last updated: 2026-07-17 00:28 (Asia/Jakarta)

## Purpose
This document is the cross-system tracker for code paths that move records, sessions, jobs, or UI workflows from one state to another. Use it before changing auth, dashboard actions, OCR, Premium, outreach, claims, publication, or migration code so edge cases are considered deliberately instead of rediscovered in production.

## Audit Method
- Search for state fields, status enums, timestamp-only state markers, queue tables, and browser state containers.
- Map each transition owner to the route/module that performs the write.
- Identify where a retry, duplicate click, expired token, stale browser state, missing secret, quota limit, or partial failure could leave the system unclear.
- Record the next concrete fix in this file, then update the row as fixes land.

## Scenario Matrix
Apply this checklist to every high-risk stateful change.

| Scenario | What to check |
| --- | --- |
| Happy path | The intended transition writes the expected state, timestamp, actor, audit/event row, and user-visible result. |
| Already done | Repeating the same action is idempotent or returns a clear "already handled" result. |
| Backward move | Moving to an earlier stage has an intentional policy for preserving or clearing later timestamps. |
| Stale session/token | Expired Clerk sessions, OAuth callbacks, Google refresh tokens, and old scopes show actionable recovery copy. |
| Missing permission | Staff/admin-only actions fail with a role-aware response, not an empty 401/403/405 or raw integration error. |
| Partial failure | External API success with local write failure, or local write success with external API failure, is traceable and retryable. |
| Duplicate click/concurrency | Two tabs, two staff members, or repeated network retries do not create duplicate orders, contacts, claims, jobs, or publish requests. |
| Quota/rate limit | Provider limits, worker caps, scheduler delays, email limits, and D1/R2 size constraints pause with clear next actions. |
| Old data shape | Rows created before a migration or before R2 text storage still render and can be reviewed safely. |
| Terminal state | Completed, cancelled, rejected, failed, expired, burned, and no-text states have explicit allowed recovery paths. |

## Working Rules For Fixes
- Prefer table-backed state transitions over state inferred only from UI flags.
- For every transition write, capture actor, timestamp, source, and an audit/event row when the action affects business data.
- For every external integration state, include a retry or reconnect path in the UI before exposing the feature to staff.
- For every queue or batch state, define idempotent retry behavior and terminal recovery behavior.
- For every reviewed data suggestion, show the source proof and the exact field-level basis before approval.
- For every migration/cleanup state, keep the operation replayable and verify remote schema/data before removing compatibility paths.
