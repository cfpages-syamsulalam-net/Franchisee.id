# Session 2026-07-10 18:46 (Asia/Jakarta)

## Focus
- Diagnose scheduler OCR batches that became `failed` without processing.
- Fix the observed production issue where retry/start created QStash batches but progress stopped.

## Findings
- Production `operation_events` showed QStash did reach `/ocr-worker`.
- Worker runs stopped at `used_today=25/25`, so the internal default `OCR_WORKER_DAILY_CAP` of 25 prevented the intended 100-job batch from draining.
- Because no further progress was written after the cap, dashboard refresh later marked the batch overdue/failed, making it look like QStash did not work.

## Changes
- Raised default worker daily cap to 100 counted units/day.
- When the worker cap is reached for a scoped batch, the batch is refreshed as `paused_quota` with a clear message instead of later being marked failed by scheduler-overdue logic.
- New/retried scheduler batches now immediately process one dashboard-scoped chunk (`run_ocr_jobs` with `batch_id`) so the click gives visible progress before the delayed QStash trigger arrives.
- `Jalankan OCR` resumes an unfinished failed/paused scheduler batch before creating a new one.
- Suggestion 81 was implemented: OCR dashboard read model now returns worker cap/used/remaining/reset state, Eksekusi Job renders it as a cap chip, and provider quota exhaustion messages suggest activating another provider or waiting for quota reset.
- OCR job rows now show the `Tanpa teks` confirmation action for `needs_review` rows, not only `failed` rows, so admin-reviewed low/no-text brochure pages can be resolved.
- OCR worker cap chip now shows a live local countdown to reset, for example `Silakan tunggu 9 jam 36 menit 14 detik lagi`.
- `js/dashboard-ocr.js` was re-audited after the worker-cap changes and refactored again: worker-cap chip rendering and reset countdown formatting now live in `js/dashboard-ocr-worker.js`, while `dashboard-ocr.js` stays as the OCR coordinator/timer/action facade.
