## D1-To-Static Publish Decision
- D1 changes do not automatically trigger Cloudflare Pages builds.
- Static HTML reflects D1 only after `pnpm run build:astro` runs during a new Pages deployment.
- Updated strategy comparison lives in `docs/architecture/D1_STATIC_PUBLISH_STRATEGY.md`.
- Target mechanism:
  - D1 mutation endpoints enqueue `site_rebuild_requests`.
  - Safe baseline is twice-daily scheduled publishing at 09:00 and 21:00 Asia/Jakarta.
  - Preferred upgrade after queue tables exist is GitHub Actions polling every 30 minutes.
  - The poller exits quickly if D1 has no pending public-page work.
  - If D1 is dirty, GitHub Actions first calls the Cloudflare Pages Deploy Hook when publish guardrails allow it, so GitHub handles checking and Cloudflare handles content builds.
  - GitHub direct `dist/` deploy with Wrangler remains the fallback if Cloudflare Pages build quota becomes constrained.
  - The poller must not commit generated output back to `main`, because the connected Cloudflare Pages Git integration can trigger an extra build from that commit.
  - The debounce/cooldown policy becomes the poll window: all edits before the next poll are batched into one build.
  - An authenticated admin-only manual trigger may publish urgent changes before the next window.
  - Successful deploys mark requests as deployed.
- `AUDIT.md` now tracks this as a planned implementation phase.
